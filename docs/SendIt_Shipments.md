## 1. Сторінка /shipments

### 1.1 Банери залежно від плану

Банер відображається над таблицею і залежить від поточного стану підключених операторів та плану підписки.

| Стан | Банер |
|------|-------|
| Жодного оператора не підключено | "Підключіть поштового оператора щоб бачити свої відправлення" + кнопка "Підключити" → `/profile` |
| FREE план (1 оператор підключено) | "Хочете відстежувати посилки з кількох поштових сервісів? Перейдіть на платний план" + кнопка "Переглянути плани" |
| PRO план (підключено 2+ операторів) | "Хочете необмежену кількість поштових сервісів? Переходьте на BUSINESS" + кнопка "Дізнатись більше" |
| BUSINESS план | Рекламний банер спецпропозицій (якщо є). Якщо немає жодних пропозицій — банер не відображається |

Банери не блокують таблицю — користувач може закрити банер або проігнорувати його.

---

### 1.2 Фільтри таблиці

Фільтри розташовані над таблицею. Основні фільтри:

- **Оператор** — вибір з підключених операторів (НП, Укрпошта, Мост тощо) + окрема опція "Драфт" (без оператора)
- **Статус** — нормалізовані статуси SendIt (див. розділ 5) + `DRAFT`
- **TTN** — пошук по трекінг-номеру
- **Діапазон дат** — дата створення посилки (від — до)
- **Діапазон вартості** — оціночна вартість (від — до)

> Повний перелік фільтрів буде розширюватись в процесі інтеграції операторів, оскільки кожен оператор може надавати додаткові поля для фільтрації.

---

### 1.3 Колонки таблиці

| Колонка | Опис |
|---------|------|
| TTN | Трекінг-номер. Для `DRAFT` — відсутній, відображається "—" |
| Оператор | Бейдж з логотипом оператора. Для `DRAFT` — бейдж "Чернетка" |
| Статус | Кольоровий бейдж нормалізованого статусу |
| Отримувач | Ім'я або назва організації отримувача |
| Дата створення | Дата і час створення посилки |
| Вартість | Оціночна вартість. Якщо відсутня — "—" |
| Дії | Кнопки дій (див. 1.4) |

Пагінація. Сортування за замовчуванням — найновіші зверху.

---

### 1.4 Дії в рядку таблиці

Кожен рядок таблиці містить набір дій. Доступність дій залежить від статусу посилки:

| Дія | DRAFT | CREATED / PREPARING | IN_TRANSIT та далі |
|-----|-------|--------------------|--------------------|
| **Переглянути** (модалка) | ✅ | ✅ | ✅ |
| **Редагувати** | ✅ | ✅ | ❌ |
| **Скасувати** | ✅ | ✅ | ❌ |
| **Дублювати** | ✅ | ✅ | ✅ |

> Точні статуси при яких доступне редагування і скасування визначаються окремо для кожного оператора в процесі інтеграції, оскільки кожен оператор має свої бізнес-правила. `DRAFT` завжди редагується і видаляється — він існує тільки локально.

---

### 1.5 Кнопки над таблицею

```
[Створити посилку]   [Фільтри ▼]   [Пошук по TTN...]
```

Кнопка **"Створити посилку"** — завжди доступна. Якщо жоден оператор не підключений — редірект на `/profile` з toast (guard).

## 2. Модалка перегляду і сторінка деталей

### 2.1 Модалка стислого перегляду

Відкривається при кліку на будь-який рядок таблиці. Містить стислу інформацію про посилку:

- TTN і бейдж оператора
- Нормалізований статус + оригінальний статус від оператора
- Дані отримувача (ім'я, телефон, адреса/відділення)
- Оціночна вартість і вага
- Дата створення

**Кнопки в модалці:**

```
[Детальніше]   [Дублювати]   [Редагувати]   [Закрити]
```

- **Детальніше** → перехід на `/shipments/:id`
- **Дублювати** → відкриває форму створення з передзаповненими даними цієї посилки
- **Редагувати** → відображається тільки якщо статус дозволяє редагування
- Кнопки `Редагувати` і `Дублювати` можуть бути приховані залежно від статусу

---

### 2.2 Сторінка деталей /shipments/:id

Окрема сторінка з повною інформацією про відправлення.

**Блоки на сторінці:**

**Блок 1 — Загальна інформація:**
- TTN, оператор, нормалізований статус, оригінальний статус від оператора
- Дата створення, дата останнього оновлення статусу

**Блок 2 — Дані отримувача:**
- Ім'я / назва організації
- Телефон, email (якщо є)
- Адреса доставки або номер відділення

**Блок 3 — Параметри посилки:**
- Вага, розміри
- Тип відправлення
- Оціночна вартість
- Додаткові поля специфічні для оператора

**Блок 4 — Історія статусів:**
- Хронологічний список змін статусу (підтягується з API оператора)
- Дата і час кожної зміни
- Оригінальна назва статусу від оператора

**Кнопки на сторінці:**
```
[← Назад до списку]   [Дублювати]   [Редагувати]   [Скасувати]
```

Доступність кнопок — за тими ж правилами що і в таблиці (залежно від статусу).

## 3. Створення посилки

### 3.1 Guard — перевірка перед відкриттям форми

Перед відкриттям форми створення система перевіряє наявність підключених операторів:

```
Клік "Створити посилку"
        │
        ▼
Є підключені оператори?
        │
   ні ──┴── так
   │              │
   ▼              ▼
Редірект на    Відкрити форму
/profile       створення
+ toast
```

---

### 3.2 Степер форми створення

Форма розбита на кроки. Поля на кроці 3 є **динамічними** — залежать від обраного оператора.

**Крок 1 — Вибір оператора:**
- Список тільки підключених операторів (активні `UserPostalConnection`)
- Вибір одного оператора для цього відправлення

**Крок 1.5 — Вибір шаблону (опційно):**
- Dropdown з шаблонами для обраного оператора
- Вибір шаблону → автозаповнює поля кроків 2 і 3
- Можна пропустити і заповнити вручну

**Крок 2 — Дані отримувача:**
- Вибір з адресної книги (`/recipients`) або заповнення вручну
- При виборі з адресної книги — автозаповнення полів
- Поля: ім'я/назва, телефон, email (опційно), адреса або відділення

**Крок 3 — Параметри посилки:**
- Динамічні поля залежно від оператора
- Загальні поля: вага, тип відправлення, оціночна вартість
- Специфічні поля оператора (наприклад тип доставки для НП, тарифна зона тощо)

**Крок 4 — Підтвердження:**
- Preview всіх введених даних
- Кнопки: **"Створити"** / **"Зберегти як драфт"** / **"Зберегти як шаблон"** / **"Скасувати"**

---

### 3.3 Кейс — відсутнє підключення до пошти

Якщо користувач починає заповнювати форму але з'єднання з обраним оператором відсутнє або недійсне (наприклад API ключ застарів):

```
Спроба створити посилку
        │
        ▼
З'єднання з оператором відсутнє / недійсне
        │
        ▼
Показати повідомлення про помилку з'єднання
        │
        ▼
Запропонувати варіанти:
   ├── [Зберегти як драфт]    → зберегти локально, завершити пізніше
   ├── [Зберегти як шаблон]   → зберегти пресет форми для майбутнього використання
   └── [Скасувати]            → не зберігати нічого
```

---

### 3.4 Успішне створення посилки

Якщо підключення активне і користувач натиснув **"Створити"**:

1. SendIt надсилає запит до API обраного оператора
2. Оператор створює посилку на своєму боці і повертає TTN та початковий статус
3. SendIt зберігає метадані локально (TTN, оператор, нормалізований статус)
4. Користувач бачить toast: *"Посилку успішно створено"*
5. Редірект на `/shipments` або на `/shipments/:id` нової посилки

## 4. Драфт vs Шаблон

### 4.1 Різниця

| | Драфт | Шаблон |
|--|-------|--------|
| **Що це** | Незавершена конкретна посилка | Пресет для створення нових посилок |
| **Де зберігається** | Локально в БД SendIt | Локально в БД SendIt |
| **Дані** | Конкретні дані отримувача, вартість тощо | Типові/повторювані дані без прив'язки до конкретного відправлення |
| **Застосування** | Завершити і відправити пізніше | Швидко створити нову посилку з передзаповненими полями |
| **TTN** | Відсутній (не відправлено оператору) | Відсутній |
| **Видалення** | Можна видалити | Можна видалити з `/templates` |
| **Відображення** | В таблиці `/shipments` зі статусом `DRAFT` | В списку `/templates` |

---

### 4.2 Збереження драфту

Драфт зберігається в таблиці `ShipmentDraft` (окрема від шаблонів):

- Зберігає всі введені дані форми у форматі `Json`
- Прив'язаний до `userId` і `postalServiceId` (якщо оператор вже обраний)
- `postalServiceId` може бути `null` якщо оператор не обраний
- Відображається в таблиці `/shipments` з бейджем "Чернетка" і статусом `DRAFT`
- При відкритті драфту — форма створення відкривається з передзаповненими даними

---

### 4.3 Автозбереження (опційно)

Варто розглянути автозбереження форми кожні N секунд у локальний стан або як драфт. Це запобігає втраті даних при випадковому закритті вкладки.

> Реалізація автозбереження — на розсуд в процесі розробки фронтенду.

## 5. Система статусів

### 5.1 Проблема

Кожен поштовий оператор має власну номенклатуру статусів. Вони можуть означати одне й те саме але мати різні назви:

```
НП:        "Відправлення в дорозі"
Укрпошта:  "В дорозі до одержувача"
Мост:      "Транспортується"
```

Фільтрація і логіка SendIt не можуть покладатись на оригінальні назви.

---

### 5.2 Двошарова система статусів

SendIt зберігає два статуси для кожної посилки:

| Поле | Тип | Опис |
|------|-----|------|
| `rawStatus` | `String` | Оригінальний статус від оператора (те що повернув API) |
| `normalizedStatus` | `ShipmentStatus` (enum) | Нормалізований статус SendIt для фільтрації і відображення |

В таблиці та фільтрах використовується `normalizedStatus`. В деталях посилки відображаються обидва — нормалізований і оригінальний.

---

### 5.3 Нормалізовані статуси SendIt

| Статус | Опис |
|--------|------|
| `DRAFT` | Чернетка. Існує тільки локально, не відправлена жодному оператору |
| `CREATED` | Посилка створена на стороні оператора, очікує прийому |
| `PREPARING` | Готується до відправки (прийнята оператором, ще не у транзиті) |
| `IN_TRANSIT` | В дорозі — передана на транспортування |
| `DELIVERED` | Доставлена отримувачу |
| `CANCELLED` | Скасована |
| `RETURNED` | Повернута відправнику |
| `UNKNOWN` | Невідомий статус — оператор повернув статус якого немає в маппінгу |

---

### 5.4 Маппінг статусів

Маппінг реалізується як **хардкод** — окрема функція-маппер для кожного оператора в коді інтеграції.

**Приклад структури маппера:**

```typescript
// mappers/nova-poshta.mapper.ts
const novaPoshtaStatusMap: Record<string, ShipmentStatus> = {
  "Створено":                    ShipmentStatus.CREATED,
  "Прийнято":                    ShipmentStatus.PREPARING,
  "Відправлення у місті":        ShipmentStatus.PREPARING,
  "Відправлення в дорозі":       ShipmentStatus.IN_TRANSIT,
  "Прибув у місто":              ShipmentStatus.IN_TRANSIT,
  "Прибув на відділення":        ShipmentStatus.IN_TRANSIT,
  "Отримано":                    ShipmentStatus.DELIVERED,
  "Відмова отримувача":          ShipmentStatus.RETURNED,
  "Повернення відправнику":      ShipmentStatus.RETURNED,
  "Відмінено":                   ShipmentStatus.CANCELLED,
};

export function mapNovaPoshtaStatus(rawStatus: string): ShipmentStatus {
  return novaPoshtaStatusMap[rawStatus] ?? ShipmentStatus.UNKNOWN;
}
```

Аналогічні маппери створюються для кожного оператора окремо при його інтеграції.

---

### 5.5 Правила редагування і видалення по статусах

| Статус | Редагувати | Скасувати/Видалити | Примітка |
|--------|-----------|-------------------|----------|
| `DRAFT` | ✅ | ✅ | Існує тільки локально |
| `CREATED` | ✅ | ✅ | Ще не прийнята фізично |
| `PREPARING` | ✅ | ✅ | Уточнюється per оператор |
| `IN_TRANSIT` | ❌ | ❌ | Вже у транзиті |
| `DELIVERED` | ❌ | ❌ | Завершена |
| `CANCELLED` | ❌ | ❌ | Вже скасована |
| `RETURNED` | ❌ | ❌ | Завершена |
| `UNKNOWN` | ❌ | ❌ | Безпечний fallback |

> Точні правила для `CREATED` і `PREPARING` уточнюються окремо для кожного оператора в процесі інтеграції. Таблиця вище — загальне правило за замовчуванням.

---

### 5.6 Обробка UNKNOWN статусу

Якщо оператор повертає статус якого немає в маппінгу:

- `normalizedStatus` → `UNKNOWN`
- `rawStatus` → оригінальне значення від оператора (зберігається)
- В таблиці відображається бейдж "Невідомий статус" з нейтральним кольором
- Редагування і скасування недоступні (безпечний fallback)
- В деталях посилки відображається оригінальний статус від оператора
